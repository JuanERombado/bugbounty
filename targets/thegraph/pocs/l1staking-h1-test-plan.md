# L1Staking H1 Test Plan

## Hypothesis

L1 staking or delegation accounting can be reduced even if the L2 receive/callhook path fails to credit the intended L2 staking account.

## Why This Is Next

The L1 gateway code explicitly warns that allowlisted callhook transfers can lock tokens on the bridge if the L2 callhook never succeeds, and `L1Staking` uses that callhook path for stake and delegation migration.

## Local Test Shape

1. Use the existing L1/L2 transfer test setup.
2. Replace or configure the L2 staking mock so the L2 receive/callhook fails for a valid L1 staking migration message.
3. Call `transferStakeToL2` or `transferDelegationToL2`.
4. Assert whether L1 accounting is reduced and GRT is escrowed before L2 credit succeeds.
5. Check whether the user has any local recovery path in the in-scope contracts.

## Expected Safe Result

A valid message can always be retried to complete the same L2 credit, or the failure cannot occur for valid in-scope callhook data.

## Vulnerable Result

L1 stake/delegation is removed, GRT is escrowed, and the L2 staking account is not credited with no practical recovery path.

## Scope Risk

This may be rejected if the only failure condition is normal Arbitrum retryable lifecycle risk or third-party bridge behavior rather than a bug in The Graph's in-scope contracts.

## First Local Question

Can we force the project-owned L2 staking callhook to reject a message that L1Staking considers valid?

## Result So Far

Yes, if L2 staking is partial-paused.

Command run from `external/thegraph-contracts/packages/contracts-test`:

```powershell
pnpm exec hardhat test --network hardhat tests/unit/l2/l2Staking.test.ts --grep "reverts a valid L1 staking migration message"
```

Observed result:

```text
1 passing
```

This is not enough for a finding because the trigger is privileged pause state and the L2 transaction itself rolls back cleanly.

## Next Local Question

Can a normal user-controlled migration generated by `L1Staking` create a permanently failing L2 callhook without relying on governor pause, invalid bridge data, compromised bridge behavior, or third-party retryable expiration?

## Local Assessment

No non-privileged permanent-failure path was identified from the reviewed code.

- `L1Staking` emits fixed valid message codes and ABI for indexer stake and delegation migration.
- L2 indexer-stake receive has no user-controlled rejection condition after gateway, sender, code, and ABI checks pass.
- L2 delegation receive returns too-small or zero-share transfers to the delegator instead of reverting.
- The demonstrated valid-message failure requires privileged partial pause state.

## Result

Rejected as a bounty candidate for now.

This remains useful background for bridge-risk understanding, but it does not currently support an Immunefi report.
